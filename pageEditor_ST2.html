🌊 Splash Screen

The Splash Screen (splash.html) is the first page users see when they visit BLUEDROP. Its purpose is to introduce the website's brand, mission, and team members before transitioning into the main site. It does not use the shared team template, but it adheres to the agreed colour palette and typography.

🔗 Open Splash Screen  |  ✅ View Validation Evidence

HTML Structure

Key tags and purpose:
  • <div id="intro-screen"> — the fixed, full-viewport overlay container for the splash. Uses position:fixed; inset:0 to cover the screen entirely.
  • <div class="background-image"> — a visually decorative layer placed behind all content using z-index:-1. Holds the full-screen background photograph via CSS background property.
  • <div class="members-list"> — a flex container that lists team member names as styled <div class="member-tag"> badges, making the team visible at first impression.
  • <div id="site-shell"> — the wrapper for the full site (header, pages, footer), initially hidden with display:none and revealed by JavaScript once the splash ends.
  • <button class="skip-button"> — provides immediate user control to bypass the auto-redirect, satisfying both accessibility and usability requirements.

CSS Techniques

  • CSS spinner animation: Two pseudo-elements (::before and ::after) on the .spinner element create a multi-ring loading indicator using only CSS @keyframes spin and border tricks. A third inner element (.spinner-core) rotates in the opposite direction, creating a dynamic visual effect without images or GIFs.
  • Glassmorphism effect: backdrop-filter:blur() and semi-transparent rgba backgrounds are applied to the goal label, member tags, and skip button, creating a frosted-glass appearance consistent with the ocean/water theme.
  • Responsive sizing: clamp() is used on the brand title to ensure it scales elegantly from mobile to desktop without media query breakpoints.
  • Overlay gradient: A multi-stop linear-gradient with rgba transparency is layered over the background image using the #intro-screen background property, ensuring text remains readable regardless of the photo content.

JavaScript Functionality

  • Countdown timer: setInterval() decrements a counter variable every 1000ms and updates the on-screen number using document.getElementById('countNumber').textContent. When the counter reaches zero, clearInterval() stops the timer and showSite() is called.
  • showSite() function: Hides the splash overlay by setting display:'none' and reveals the site shell by setting display:'flex', then calls showPage('home') (the team's shared SPA routing function) to load the home page content.
  • Skip button: The skipSplash() function calls clearInterval() immediately to cancel the automatic timer, then calls showSite(). This avoids race conditions where both the timer and the button might try to transition simultaneously.

⚡ Action Impact Simulator (AIS)

The Action Impact Simulator (ais.html) is an interactive page that allows users to select SDG 6-related water conservation actions and see their combined impact score update in real time. The page background changes dynamically based on the user's impact level.

🔗 Open Action Impact Simulator  |  ✅ View Validation Evidence

HTML Structure

  • <main> — the primary landmark region, containing the simulator banner, impact key, feedback panel, and cards grid. Using <main> ensures screen readers identify this as the primary content area.
  • <div class="impact-panel" aria-live="polite" aria-atomic="true"> — the score and level display area. The aria-live="polite" attribute causes screen readers to announce changes to the panel each time the score updates, without interrupting the user mid-speech.
  • <div class="actions-grid" role="list"> — a container for dynamically generated action cards. The role="list" attribute is used because the grid is built by JavaScript, not a native <ul> or <ol>, so the ARIA role explicitly communicates the list semantics.
  • Action cards are <article> elements generated by JavaScript with tabindex="0" and aria-pressed attributes, making them keyboard navigable and conveying their toggle state to assistive technology.

CSS Techniques

  • Grid layout: display:grid; grid-template-columns: repeat(auto-fill, minmax(270px, 1fr)) creates a fully responsive card grid that reflows from one to three columns without any media queries.
  • Selected card state: The .action-card.selected rule applies a distinct border colour, gradient background, and box-shadow. JavaScript adds and removes the selected class — the visual feedback is therefore driven by JavaScript logic but expressed through CSS.
  • Check badge visibility: .check-badge has display:none by default and switches to display:flex only when the parent card has the .selected class, providing clear visual confirmation of selection.
  • Image hover zoom: A smooth transform:scale(1.03) on the card image creates a subtle zoom-in effect on hover, improving the interactive feel of the cards.

JavaScript Logic

  • Data-driven cards: An array of action objects (aisActions) holds all card data. The initAIS() function loops over this array using forEach and creates each card element via document.createElement() and innerHTML assignment, demonstrating DOM manipulation.
  • Set for selection tracking: A JavaScript Set (aisSelected) stores the indices of selected cards. A Set is ideal for this: has(), add(), and delete() all operate in O(1) time, and it naturally prevents duplicate selections.
  • Impact calculation: The updateAIS() function iterates over aisSelected with forEach, summing the pts value for each selected action. It then applies conditional logic to determine the impact level: Low (1–5 pts), Medium (6–12 pts), High (13+ pts).
  • JavaScript-controlled background change (spec §4.5.6): Based on the calculated impact level, updateAIS() assigns a gradient string directly to element.style.background. This is a JavaScript DOM manipulation, not a CSS-only trick, fully satisfying the coursework requirement.
  • Keyboard accessibility: A keydown event listener on each card triggers the toggle function on Enter or Space, allowing full keyboard-only interaction without a mouse.

📄 Content Page

My individual content page (content_ST2.html) focuses on water conservation, covering why freshwater is under threat, daily waste, global impact, and actions students can take. It uses the team's shared header/footer template and global CSS, with additional page-specific embedded CSS.

🔗 Open Content Page  |  ✅ View Validation Evidence

Key HTML elements used:
  • <section> elements with unique id attributes divide the page into five clearly named content areas, supporting both the internal navigation and screen-reader landmark structure.
  • <figure> and <figcaption> are used for all images, ensuring each photograph has a semantically associated caption that adds descriptive context beyond the alt attribute.
  • <nav class="toc"> provides the internal table of contents — an ordered list of anchor links pointing to each section's id, satisfying the internal navigation requirement (spec §4.10.4).
  • <a href="#top" class="go-top"> is a fixed-position "Back to Top" button implemented using position:fixed CSS, with no JavaScript required (spec §4.10.6).
  • scroll-margin-top:90px is applied to each content section so that when an internal link is clicked, the target section is not obscured by the sticky header.

🔧 Challenges and Lessons Learned

Challenge 1 — Splash Screen transition without page reload

The initial approach used a <meta http-equiv="refresh"> tag (as suggested in the lecture notes), but the team decided to build a single-page application (SPA) structure so that the shared header would only load once. This meant the splash needed to transition into the site without a full page reload. I resolved this by using JavaScript to toggle display between the splash overlay and the site shell, which was a cleaner approach and gave us more control over the transition timing.

Challenge 2 — AIS background change via JavaScript only

The coursework specification required the background change to be controlled by JavaScript, not CSS-only tricks. My initial draft used CSS classes and :nth-child selectors, which would not have met the requirement. I refactored to store gradient strings in a JavaScript object (var BG = {...}) and assign them to element.style.background directly inside the updateAIS() function. This clearly demonstrates JavaScript-driven DOM manipulation.

Challenge 3 — Keyboard accessibility for dynamically generated cards

Because the action cards are built by JavaScript, standard HTML focus behaviour did not apply by default. I learned to add tabindex="0" to each card element and attach keydown event listeners for Enter and Space, mirroring the behaviour of native interactive elements. This taught me that custom interactive components need explicit accessibility handling that native HTML provides for free.

Lessons Learned

This project deepened my understanding of the relationship between HTML, CSS, and JavaScript. I learned that writing valid, semantic HTML is not a constraint — it actually simplifies both accessibility and styling. I also improved my ability to structure JavaScript into reusable functions (initAIS, updateAIS, toggleCard), which made the code much easier to debug and read.

♿ Accessibility Considerations

Accessibility was considered throughout the implementation of all three pages. The following techniques were applied:

  • Semantic HTML: Landmark elements (<header>, <nav>, <main>, <footer>, <section>) provide structural information to screen readers, enabling efficient navigation between page regions without visual reference.
  • Alt text on all images: Every image across all three pages includes a descriptive alt attribute. Decorative elements (such as the emoji in the hero banner) use aria-hidden="true" to prevent them from being announced unnecessarily.
  • ARIA live region: The AIS impact panel uses aria-live="polite" so that score updates are announced to screen-reader users without disrupting their workflow.
  • Keyboard navigation: AIS cards support Enter and Space key toggling. The skip button on the splash screen is a native <button>, which is keyboard-focusable by default.
  • Colour contrast: All text uses colours from the team's design tokens with sufficient contrast ratios. The white text on the dark gradient hero banners meets WCAG AA contrast requirements.
  • Focus states: Native browser focus outlines are preserved (not removed with outline:none), ensuring keyboard users can always see which element is focused.
